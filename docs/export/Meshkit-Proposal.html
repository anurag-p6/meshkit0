1. Introduction

1.1 Purpose of This Document

This document is the Meshkit project proposal: the full case for what Meshkit is, why it should exist, and how it is built. Meshkit is not a side feature on top of something else, and PPT is not a standalone proposal. Meshkit is the product; PPT-gated IPFS access is how that product works.

The proposal covers:

  • What Meshkit is: an IPFS storage SDK for React Native, Capacitor, and Node.js (§3)
  • Why Meshkit matters: decentralized upload, retrieve, and pin against Kubo, with multi-node failover and mobile support (§2, §4)
  • How access works: holders of Park Pro Token (PPT) on Arbitrum use Meshkit APIs; non-holders are routed to Uniswap V3 (§2, §6-7)
  • How it is implemented: architecture, APIs, errors, platforms, security (§8-14)
  • How it ships: roadmap, open decisions, documentation skill (§16-18)

1.2 What This Proposal Delivers

Meshkit is proposed as a complete SDK product with three integrated pillars:

Pillar What Meshkit delivers
IPFS storage SDK Meshkit.init, upload, retrieve, pin against Kubo RPC; multi-node failover
PPT token-gated access Hold-to-use gate via ERC-20 balanceOf on Arbitrum before every storage operation
Mobile & hybrid support @ipfs-meshkit/react-native, @ipfs-meshkit/capacitor, polyfills for real devices

PPT is the access credential. Meshkit is the product that consumes and enforces it while delivering IPFS storage to apps.

1.3 Scope

In scope for Meshkit (this proposal):

  • Core SDK: Meshkit facade, Kubo client, health checks, node failover (@ipfs-meshkit/core)
  • PPT gate: wallet balance check, TokenGateError, Uniswap onboarding links (PPT/ARB, PPT/USDC)
  • Platform packages: React Native polyfills, Capacitor adapter
  • Technical documentation and a Documentation skill blueprint

Out of scope for v1 (unless promoted in §18):

  • On-chain micropayments per upload/retrieve
  • Kubo-native authorization plugins
  • Multi-chain PPT deployment
  • Flutter adapter (listed as future work)

2. Meshkit Value Proposition

This section states why Meshkit should be built and how PPT fits into the product, not as an afterthought, but as the designed access model for IPFS storage.

2.1 Value Proposition (One Sentence)

Meshkit is the SDK for PPT-gated IPFS storage: hold PPT, use Meshkit, store and access data on IPFS.

Meshkit gives developers one integration for decentralized file storage on mobile and web. Park Pro Token (PPT) is the on-chain credential that unlocks it: upload, retrieve, and pin against Kubo nodes for wallets that hold enough PPT.

2.2 The Problem Meshkit Solves

Developers building invoice apps, document backup, attachments, and other products need decentralized storage without embedding IPFS inside the app. Today they face two gaps Meshkit is proposed to close:

Gap Without Meshkit With Meshkit
IPFS integration Raw Kubo RPC, no failover, poor mobile support Typed SDK: upload / retrieve / pin, multi-node failover, RN/Capacitor packages
Access control Open Kubo URLs or custom auth per app PPT hold-to-use gate built into every storage call
User onboarding No standard path when access is denied TokenGateError + Uniswap V3 (PPT/ARB, PPT/USDC)

Meshkit utility definition:
Meshkit is how apps and users perform IPFS storage operations: upload, retrieve, and pin on Kubo nodes, with PPT on Arbitrum as the required access credential.

PPT does not replace Meshkit. Meshkit is the product; PPT is what grants access to it.

2.3 Why PPT Powers Meshkit Access

Meshkit is designed around PPT as the access layer. Park Pro Token is an ERC-20 on Arbitrum; Meshkit reads balanceOf(wallet) before each storage operation.

Property Role in Meshkit

Hold-to-use Users keep PPT; no per-upload gas in v1
DEX liquidity Uniswap V3 (PPT/ARB, PPT/USDC) is the onboarding path when gate fails

2.6 What Meshkit Storage Looks Like in Practice

For end users and developers, "token-gated IPFS storage" is not abstract. It maps directly to Meshkit methods:

User intent Meshkit API What happens on IPFS PPT required?
Save a file mk.upload(bytes) Content added to Kubo; CID returned Yes
Open a file by CID mk.retrieve(cid) Bytes fetched from Kubo Yes
Keep a file pinned mk.pin(cid) CID pinned on node (not GC'd) Yes
Check node is up Meshkit.init({ nodes }) Health check only No (diagnostics)

Example (invoice app): A freelancer app uses Meshkit to upload PDF invoices to IPFS. The freelancer connects a wallet. If they hold PPT, upload succeeds and returns a CID they can share. If not, the app shows Meshkit's TokenGateError and a link to buy PPT, then they retry.

3. What Meshkit Is

Meshkit is proposed as a TypeScript SDK that connects apps to Kubo IPFS nodes and enforces PPT-gated access before storage operations. This section describes the product architecture.

3.1 Overview

Meshkit targets mobile and hybrid frameworks (React Native, Capacitor/Ionic) and Node.js.

It does not embed an IPFS node in the app. It speaks HTTP to a running Kubo (go-ipfs) daemon on the developer's PC, LAN, or VPS. Data lives on that node. The SDK is a typed client over Kubo's RPC API (kubo-rpc-client). The public facade is the Meshkit class in @ipfs-meshkit/core.

Proposed access model: before upload, retrieve, or pin, Meshkit checks PPT balance on Arbitrum. That gate is part of the product design, not an optional plugin.

3.2 Package Structure

Package Role
@ipfs-meshkit/core Meshkit facade, createMeshkitClient, failover, health checks
@ipfs-meshkit/react-native Re-exports core + polyfills entry for RN runtime gaps
@ipfs-meshkit/capacitor Capacitor plugin scaffold + web fallback

3.3 Current Initialization Flow

Meshkit.init(options) accepts:

  • nodes: string[]: Kubo RPC URLs in priority order (primary first, others for failover)
  • headers?: Record<string, string>: optional HTTP headers (e.g. API auth on the node)

On init, the SDK:

  1. Creates one MeshkitClient per URL via createMeshkitClient
  2. Runs healthCheck() on each (calls ipfs.id())
  3. Drops unreachable nodes
  4. Throws MeshkitError if zero nodes are healthy
  5. Returns a Meshkit instance with activeNodes listing survivors

There is no wallet, no blockchain RPC, and no identity beyond whatever the Kubo endpoint enforces (often nothing on local dev).

3.4 Current Operation Semantics

Each public method on Meshkit uses withFailover: try nodes in order until one succeeds.

Method Kubo RPC Behavior
upload(data: Uint8Array) ipfs.add(data, { pin: false }) Returns CID string; does not pin by default
retrieve(cid: string) ipfs.cat(cid) Streams chunks into a single Uint8Array
pin(cid: string) ipfs.pin.add(cid) Pins CID on the node
healthCheck() (per client) ipfs.id() Used only at init

3.5 Authorization Model (Proposed)

Meshkit proposes two layers of access:

Layer Mechanism
Network App must reach Kubo RPC URL (optional HTTP headers)
PPT (product gate) Wallet must hold ≥ minimum PPT on Arbitrum before storage ops

Without the PPT layer, any client with the Kubo URL can use storage: which is why PPT gating is core to the Meshkit proposal.

3.6 Deployment Patterns

  1. Local development: ipfs daemon on the dev machine; app uses 127.0.0.1:5001 (Node) or LAN IP (physical mobile device)
  2. Single VPS: one Kubo instance; app points to https://node.example.com:5001
  3. Multi-node failover: primary local + backup VPS; withFailover masks transient failures

In all patterns, Meshkit adds wallet-bound PPT checks before storage operations. Network reachability alone is not enough.

4. Problem Statement

4.1 Why Meshkit Needs to Exist

Mobile and web developers lack a purpose-built SDK for IPFS storage that also solves who may store data. Meshkit is proposed to address both:

  1. IPFS integration gap: Raw Kubo RPC is low-level; mobile needs polyfills; production needs failover across nodes.
  2. Access control gap: Shared Kubo URLs are not user-level permissions; each app should not rebuild wallet + token logic.
  3. Onboarding gap: When a user cannot store data, there must be a standard path to qualify (hold PPT, buy on Uniswap).

4.2 Technical Problems Meshkit Addresses

  1. No verifiable user entitlement: Apps cannot ask: "Is this user allowed to store on IPFS?" without custom infrastructure.

  2. No scalable access policy: Kubo operators cannot map users to quotas; header auth is per-deployment and not portable across React Native, Capacitor, and Node.

  3. No standardized denial path: Blocked users need a single UX: hold PPT, or acquire it on Uniswap V3.

4.3 Why PPT Is Part of the Meshkit Proposal

PPT is the chosen access credential for Meshkit because ERC-20 balanceOf on Arbitrum is verifiable, portable, and liquid on Uniswap: without per-operation gas in v1.

4.4 Stakeholders

End user Decentralized storage in apps; clear path to get PPT

4.5 Formal Problem Statement

Meshkit shall provide a TypeScript SDK for IPFS storage (upload, retrieve, pin) against Kubo nodes with multi-node failover and mobile platform support, requiring wallets to hold a minimum balance of PPT on Arbitrum before storage operations, and directing ineligible users to acquire PPT on Uniswap V3.

5. Goals, Constraints, and Non-Goals

5.1 Goals

G1: Entitlement before IO
Every gated operation (upload, retrieve, pin) must await a successful token gate evaluation before any Kubo RPC is invoked.

G2: Fail closed on ambiguity
If the gate cannot determine eligibility (RPC down, wrong chain, no wallet address), the operation must not proceed to IPFS. The error must state what the user or developer should fix.

G3: Developer ergonomics
Existing Meshkit.init({ nodes }) call sites add a tokenGate block; method signatures on Meshkit stay unchanged.

G4: Actionable end-user denial
TokenGateError includes: current balance, required minimum, formatted human amounts, and links to PPT/ARB and PPT/USDC on Uniswap V3.

G5: Documented thoroughly
This proposal plus the Documentation skill (§16) enable accurate public docs without re-deriving design decisions.

5.2 Constraints

C1: Client-side enforcement in v1
The gate runs in the app process. Kubo remains unaware of PPT unless the operator adds separate middleware (out of v1 scope).

C2: Arbitrum only
PPT contract and balanceOf reads target Arbitrum One (chainId 42161) unless §18 changes this.

C3: Read-only chain interaction
v1 only reads balanceOf. No transfer, approve, or meta-transactions.

C4: Backward compatibility
tokenGate is optional. Omitting it or setting enabled: false preserves today's behavior.

5.3 Non-Goals (v1)

  • Burning or transferring PPT per operation
  • NFT-based gating (ERC-721/1155)
  • Staking or time-locked balances
  • Replacing Kubo or embedding IPFS in-process
  • Guaranteed server-side enforcement without operator action
  • Wallet private key management inside Meshkit

6. Token-Gated Access Model

6.1 Gate Type: Threshold Balance (Hold-to-Use)

Meshkit implements a static threshold gate as part of the product:

eligible(wallet) := balanceOf(PPT, wallet) >= minimumBalance
  • minimumBalance is in token base units (wei-style: 10n ** 18n for 1 PPT if decimals = 18)
  • No expiry: balance checked at operation time (with optional short TTL cache per §9.4)
  • Spending PPT on Uniswap reduces balance; user may become ineligible until they hold enough again

6.2 Gating Granularity

Operation Gated? Rationale
upload Yes Primary cost: ingesting data onto IPFS
retrieve Yes Prevents free egress via shared Kubo URL
pin Yes Pinning consumes node storage
Meshkit.init No Allow node health diagnosis without tokens
healthCheck No Internal; used at init
createMeshkitClient (low-level) Configurable Bypass for advanced operators; default document as ungated escape hatch

6.3 Decision Flow

flowchart TD
  START([App calls upload / retrieve / pin]) --> ENABLED{tokenGate.enabled?}
  ENABLED -->|false| IPFS[Execute Kubo RPC via failover]
  ENABLED -->|true| WALLET{Wallet address available?}
  WALLET -->|no| WE[Throw WalletError]
  WALLET -->|yes| CHAIN{On Arbitrum?}
  CHAIN -->|no| WE
  CHAIN -->|yes| BAL[Read PPT balanceOf via RPC]
  BAL --> RPCOK{RPC success?}
  RPCOK -->|no| WE
  RPCOK -->|yes| CMP{balance >= minimum?}
  CMP -->|yes| IPFS
  CMP -->|no| TGE[Throw TokenGateError + Uniswap links]
  IPFS --> DONE([Return result])
  TGE --> DENY([Propagate to app UI])
  WE --> DENY

6.4 Relationship to the Reference Diagram

The diagram encodes the same logic from the user journey perspective:

  1. User opens an app using Meshkit
  2. User attempts an IPFS action
  3. System asks: "Holds Park Pro Tokens?"
    • Yes → action proceeds against IPFS
    • No → user is instructed to get PPT on Uniswap V3 (PPT/ARB or PPT/USDC), then retry

The SDK automates step 3; the app's job is to surface errors and deep-link to Uniswap when needed.

6.5 What "Perform Any Action" Means

For v1, "any action" is defined as all methods on Meshkit that mutate or read IPFS content through Kubo:

  • upload, retrieve, pin

Future methods (e.g. unpin, ls, stat) should inherit the same gate via a single internal assertTokenGate() hook so policy stays centralized.

7.3 Call Order Invariant

For every gated operation, the SDK must guarantee:

1. assertTokenGate()   // may throw TokenGateError | WalletError
2. withFailover(...)   // may throw MeshkitError

Never reverse this order. Hitting Kubo before the gate would violate G1.

9. Detailed Component Design

8.1 TokenGateConfig

interface TokenGateConfig {
  /** Master switch. false = skip all gate logic (dev/test). */
  enabled: boolean;

  /** PPT contract on Arbitrum. */
  contractAddress: `0x${string}`;

  /** Must be 42161 for canonical PPT unless multi-chain is added later. */
  chainId: number;

  /**
   * Wallet address to check. Typically the connected account.
   * Meshkit does not connect wallets; the app supplies this.
   */
  walletAddress: `0x${string}`;

  /** Minimum balance in token base units (not human-readable floats). */
  minimumBalance: bigint;

  /** Token decimals for display; if omitted, fetch once from contract. */
  decimals?: number;

  /** Arbitrum HTTP RPC URL for eth_call. */
  rpcUrl: string;

  /** Uniswap deep links shown in TokenGateError. */
  uniswap: {
    pptArb: string;
    pptUsdc: string;
  };

  /** Optional: balance cache TTL in milliseconds. Default 30_000. */
  balanceCacheTtlMs?: number;
}

Design choice: walletAddress is passed in config (or via a getter) rather than Meshkit owning WalletConnect. Keeps core free of UI dependencies and matches how mobile apps already manage wallets.

8.2 Token Gate Service

createTokenGate(config: TokenGateConfig): TokenGate

Method Behavior
assertEligible() Throws if not eligible; no-op if enabled: false
getBalance() Returns current bigint balance (uses cache)
isEligible() Non-throwing boolean
invalidateCache() Force next read on-chain (call after user returns from Uniswap)

Balance read implementation:

  1. eth_call to balanceOf(walletAddress) with PPT ABI
  2. Decode uint256 return
  3. Compare >= minimumBalance

Use a library (recommended: viem createPublicClient + readContract) for ABI encoding and RPC handling.

8.3 Integration into Meshkit

class Meshkit {
  private readonly tokenGate: TokenGate | null;

  upload(data: Uint8Array): Promise<string> {
    return this.runGated((client) => client.upload(data));
  }

  private async runGated<T>(
    op: (client: MeshkitClient) => Promise<T>,
  ): Promise<T> {
    await this.tokenGate?.assertEligible();
    return withFailover(this.clients, op);
  }
}

Meshkit.init constructs TokenGate when options.tokenGate is present and enabled !== false.

8.4 Balance Cache

Concern Approach
RPC rate limits Cache last (balance, timestamp) per wallet+contract
Stale eligibility TTL default 30s; document invalidateCache() after swap
User spends PPT mid-session May pass gate then drop below threshold before next op; next op fails

Cache is an optimization, not a security boundary (client-side gate is already bypassable; see §14).

8.5 Chain Validation

Before balanceOf, verify:

  • eth_chainId matches config.chainId
  • If mismatch → WalletError with message to switch to Arbitrum

Optional: wallet_getChainId when using an injected provider passed from the app.

  Meshkit->>Health: filterHealthy(clients)
  loop each node
    Health->>Kubo: ipfs.id()
    Kubo-->>Health: ok / fail
  end
  Health-->>Meshkit: healthy clients
  Meshkit-->>App: Meshkit instance

Init does not require PPT. Developers can verify Kubo connectivity before wallet setup.

9.2 Gated Upload

sequenceDiagram
  participant App
  participant Meshkit
  participant Gate
  participant Arbitrum
  participant Kubo

  App->>Meshkit: upload(bytes)
  Meshkit->>Gate: assertEligible()
  Gate->>Arbitrum: eth_call balanceOf(wallet)
  Arbitrum-->>Gate: balance

  alt balance < minimum
    Gate-->>App: throw TokenGateError
  else balance >= minimum
    Gate-->>Meshkit: ok
    Meshkit->>Kubo: ipfs.add (failover)
    Kubo-->>Meshkit: CID
    Meshkit-->>App: CID
  end

9.3 Gated Retrieve

Same as upload through assertEligible(), then ipfs.cat(cid) via failover.

9.4 Gated Pin

Same gate, then ipfs.pin.add(cid).

9.5 Denied User Path (Product Flow)

sequenceDiagram
  participant User
  participant App
  participant Meshkit
  participant Uniswap

  User->>App: Upload invoice
  App->>Meshkit: upload(data)
  Meshkit-->>App: TokenGateError
  App->>User: Show balance + Get PPT
  User->>Uniswap: Swap ARB/USDC → PPT
  User->>App: Retry upload
  App->>Meshkit: invalidateCache + upload
  Meshkit-->>App: CID

11.2 Usage Example (Full)

import { Meshkit, TokenGateError } from '@ipfs-meshkit/core';

const mk = await Meshkit.init({
  nodes: ['https://ipfs.example.com:5001'],
  tokenGate: {
    enabled: true,
    contractAddress: '0xTBD',
    chainId: 42161,
    walletAddress: '0xUserWallet',
    minimumBalance: 10n ** 18n, // 1 PPT if 18 decimals
    rpcUrl: 'https://arb1.arbitrum.io/rpc',
    uniswap: {
      pptArb: 'https://app.uniswap.org/#/swap?chain=arbitrum&TBD',
      pptUsdc: 'https://app.uniswap.org/#/swap?chain=arbitrum&TBD',
    },
  },
});

try {
  const cid = await mk.upload(invoiceBytes);
} catch (err) {
  if (err instanceof TokenGateError) {
    // err.balance, err.required, err.uniswap.pptArb, err.uniswap.pptUsdc
  }
}

12. Error Handling and User Feedback

12.1 Error Types

TokenGateError

Field Type Description
message string Human-readable summary
balance bigint Current PPT balance (base units)
required bigint minimumBalance
balanceFormatted string e.g. "0.5 PPT"
requiredFormatted string e.g. "1 PPT"
walletAddress string Checked address
uniswap { pptArb, pptUsdc } Acquisition links

Example message:

Insufficient Park Pro Token (PPT) balance. You have 0 PPT but need at least 1 PPT to use IPFS through Meshkit. Get PPT on Uniswap: PPT/ARB or PPT/USDC.

WalletError

Condition Example message
Missing walletAddress Connect a wallet to use IPFS.
Wrong chain Switch your wallet to Arbitrum (chain ID 42161).
RPC failure Could not read PPT balance. Check your network connection.

MeshkitError (existing)

Unchanged: all Kubo nodes failed. Distinct from token gating.

12.2 Error Precedence

If both gate and Kubo could fail, gate is checked first. Users without PPT should not trigger Kubo traffic.

12.3 UI Guidelines for App Developers

  1. Catch TokenGateError separately from MeshkitError
  2. Show formatted balances, not raw bigint
  3. Render two buttons: "Get PPT (ARB)" and "Get PPT (USDC)" opening uniswap links
  4. On return from browser, call invalidateCache() before retry

13. Platform Integration

13.1 Node.js / Desktop

  • Wallet via browser injection not applicable; use server-side wallet address + RPC, or WalletConnect in Electron
  • tokenGate.enabled: false for CI and unit tests

13.2 React Native

Existing requirements remain:

  • Import @ipfs-meshkit/react-native/polyfills before Meshkit
  • Use LAN IP or VPS URL, not 127.0.0.1 on device

Additional for v1:

  • Integrate a RN wallet library (e.g. WalletConnect, MetaMask SDK)
  • Obtain walletAddress and ensure Arbitrum chain before Meshkit.init or pass a getter
  • Arbitrum RPC: use a reliable provider; avoid blocking the JS thread on long eth_call

13.3 Capacitor / Ionic

  • Web: window.ethereum may be available in WebView with injected provider
  • Native: bridge to wallet app via deep link or WalletConnect
  • Capacitor plugin may later wrap gated Meshkit init; v1 can use core directly from app code

13.4 Polyfills and Chain Libraries

Token gate adds Ethereum JSON-RPC dependency. Keep it out of react-native/polyfills.ts; wallet + viem/ethers are app-level dependencies to avoid bloating apps that use enabled: false.

14. Security Model and Threat Analysis

14.1 Trust Boundaries

flowchart LR
  subgraph TrustedByIssuer["Trusted by Park Pro"]
    PPT[PPT contract on Arbitrum]
  end

  subgraph Untrusted["Untrusted client environment"]
    SDK[Meshkit SDK]
    App[App binary / JS bundle]
  end

  subgraph Operator["Node operator control"]
    Kubo[Kubo RPC]
  end

  App --> SDK
  SDK -->|read-only| PPT
  SDK -->|HTTP| Kubo

14.2 What the Gate Provides

  • Honest app path enforcement: Official SDK refuses operations for ineligible wallets
  • Consistent UX: Documented errors and onboarding
  • Policy centralization: One place to update threshold and contract address

14.3 What the Gate Does Not Provide

Threat Mitigation in v1 Stronger mitigation (future)
User patches SDK / skips gate None on client alone Signed attestations, server proxy
Leaked Kubo URL None API auth, IP allowlist, VPN
Fake PPT contract in config Document canonical address Hardcode mainnet address in production builds
RPC spoofing Use trusted RPC URL Multiple RPC sources, proof verification
Sybil with minimal PPT minimumBalance policy Higher threshold, rate limits on proxy

Conclusion: The client-side token gate is economic and UX policy, not a cryptographic access control on Kubo. Operators exposing public Kubo must add server-side controls for high-assurance deployments.

14.4 Fail-Closed Rules

Scenario Behavior
balanceOf reverts WalletError, no IPFS
RPC timeout WalletError, no IPFS
walletAddress zero address WalletError, no IPFS
enabled: true but missing config fields Throw at init, not at first upload
